## 摘要
多跳问答（QA）需要跨多个文档进行推理，然而现有的检索增强生成（RAG）方法通常通过需要额外在线处理的图方法或迭代式多步推理来解决这一问题。我们提出了 IndexRAG，这是一种将跨文档推理从在线推理阶段转移到离线索引阶段的新颖方法。IndexRAG 识别跨文档共享的桥接实体，并将桥接事实生成为可独立检索的单元，且无需额外的训练或微调。在三个广泛使用的多跳问答基准测试（HotpotQA、2WikiMultiHopQA、MuSiQue）上的实验表明，IndexRAG 的 F1 分数比 Naive RAG 平均提高了 4.6 个百分点，同时在推理阶段仅需单次检索和单次大语言模型（LLM）调用。当与 IRCoT 结合时，IndexRAG 在平均性能上优于包括 HippoRAG 和 FastGraphRAG 在内的所有基于图的基准模型，而其仅依赖于扁平检索。我们的代码将在论文被接收后发布。

## Abstract
Multi-hop question answering (QA) requires reasoning across multiple documents, yet existing retrieval-augmented generation (RAG) approaches address this either through graph-based methods requiring additional online processing or iterative multi-step reasoning. We present IndexRAG, a novel approach that shifts cross-document reasoning from online inference to offline indexing. IndexRAG identifies bridge entities shared across documents and generates bridging facts as independently retrievable units, requiring no additional training or fine-tuning. Experiments on three widely-used multi-hop QA benchmarks (HotpotQA, 2WikiMultiHopQA, MuSiQue) show that IndexRAG improves F1 over Naive RAG by 4.6 points on average, while requiring only single-pass retrieval and a single LLM call at inference time. When combined with IRCoT, IndexRAG outperforms all graph-based baselines on average, including HippoRAG and FastGraphRAG, while relying solely on flat retrieval. Our code will be released upon acceptance.

#### 1. 核心问题与整体含义（研究动机和背景）
传统的检索增强生成（RAG）在处理**多跳问答（Multi-hop QA）**时面临挑战，因为这类问题需要整合分布在不同文档中的碎片化信息。现有的解决方案通常分为两类：
*   **基于图的方法（如 GraphRAG, HippoRAG）：** 在推理时需要复杂的实体提取、图遍历或多次 LLM 调用，导致延迟高、成本贵。
*   **迭代式方法（如 IRCoT）：** 通过多次“检索-生成”循环来逐步寻找线索，同样增加了推理阶段的负担。

**IndexRAG 的核心动机**是将跨文档推理的压力从“在线推理阶段”前移至“离线索引阶段”。作者认为文档间的逻辑联系是客观存在的，与具体查询无关，因此可以预先计算并生成“桥接事实”，从而在推理时只需单次检索和单次 LLM 调用即可完成复杂推理。

---

#### 2. 论文提出的方法论
IndexRAG 采用两阶段的离线索引流程和一种平衡的在线推理机制：

*   **离线索引阶段：**
    1.  **阶段 1：AKU 与实体提取。** 使用 LLM 从每个文档中提取“原子知识单元”（AKU，以问答对形式呈现的事实）和相关实体。
    2.  **阶段 2：桥接事实生成（Bridging Fact Generation）。** 
        *   识别**桥接实体**：即在多个文档中共同出现的实体（过滤掉频率过高或过低的实体）。
        *   生成**桥接事实**：将共享同一实体的多个文档片段输入 LLM，要求其生成能够连接这些信息的逻辑结论。例如，文档 A 说“X 导演了电影 Y”，文档 B 说“X 出生在 Z”，IndexRAG 会生成桥接事实“电影 Y 的导演出生在 Z”。
    3.  **统一存储：** 将 AKU 和生成的桥接事实全部向量化，存入同一个扁平向量数据库。

*   **在线推理阶段：**
    *   **平衡上下文选择（Balanced Context Selection）：** 由于桥接事实通常比原始 AKU 短，容易在检索中占据前排位置。为了防止其挤占信息密度更高的原始上下文，算法设置了参数 $k_b$，限制检索结果中桥接事实的最大数量，确保上下文的丰富性。
    *   **单次生成：** 将筛选后的上下文直接喂给 LLM 生成答案。

#### 6. 论文的主要结论与发现
*   **性能领先：** 在单次 LLM 调用的方法中，IndexRAG 在三个数据集上的平均 F1 达到 51.7，显著优于 Naive RAG（47.1）和 FastGraphRAG（49.4）。
*   **强强联手：** 当 IndexRAG 与 IRCoT 结合时，其表现超越了所有基准方法（包括复杂的 HippoRAG），证明了预生成的桥接事实能有效辅助迭代推理。

#### 7. 优点
*   **推理高效：** 实现了“索引即推理”，在推理阶段保持了与简单 RAG 相同的低延迟和低成本。
*   **无需训练：** 这是一个 Training-free 的框架，不需要微调 Embedding 模型或 LLM，易于部署。
*   **即插即用：** 桥接事实生成模块可以叠加在任何现有的 RAG 系统之上。
*   **逻辑显式化：** 将隐含在文档间的逻辑链条转化为显式的可检索单元。

---

#### 8. 不足与局限
*   **索引成本：** 虽然推理省钱，但离线索引阶段需要大量 LLM 调用，对于海量文档库，索引成本和时间可能较高。
*   **实体依赖：** 桥接事实的质量高度依赖于实体提取的准确性，若实体识别有误或存在歧义，会生成错误的桥接事实。
*   **幻觉风险：** 桥接事实是由 LLM 生成的二次创作内容，存在产生幻觉或误导性关联的风险。
*   **覆盖范围：** 目前仅在英文多跳 QA 场景下验证，对于长文本理解、多模态或多语言场景的泛化能力尚待观察。
