## Abstract 

Modern agentic systems allow Large Language Model (LLM) agents to tackle complex tasks through extensive tool usage, forming structured control flows of tool selection and execution. Ex-isting security analyses often treat these control flows as ephemeral, one-off sessions, overlooking the persistent influence of memory. This paper identifies a new threat from Memory Control Flow Attacks (MCFA) that memory retrieval can dom-inate the control flow, forcing unintended tool usage even against explicit user instructions and inducing persistent behavioral deviations across tasks. To understand the impact of this vulnerabil-ity, we further design M EM FLOW , an automated evaluation framework that systematically identi-fies and quantifies MCFA across heterogeneous tasks and long interaction horizons. To evaluate MEM FLOW , we attack state-of-the-art LLMs, in-cluding GPT-5 mini, Claude Sonnet 4.5 and Gem-ini 2.5 Flash on real-world tools from two major LLM agent development frameworks, LangChain and LlamaIndex. The results show that in general over 90% trials are vulnerable to MCFA even un-der strict safety constraints, highlighting critical security risks that demand immediate attention.

As agents increasingly coordinate multiple tools to achieve a single objective, their behavior naturally induces structured control flows over tool selection and execution (Li et al., 2026). The presence of control flows over tool selection and execu-tion in LLM-powered agentic systems introduces a series of new security challenges (Debenedetti et al., 2024). Prior work has demonstrated vulnerabilities that arise from ma-nipulating agent control flow, including biased tool selection (Xu et al., 2025), reordered or omitted safety-critical work-flow steps (Jia et al., 2025), and the malicious instructions in-jection through untrusted tool -ingested content (Seetharam et al., 2025). In practice, LLM agents store critical information as memory, including user profiles (Packer et al., 2023), user preferences (Chen et al., 2024), and historical interaction context (Anwar et al., 2024). The persistent presence of memory in agentic systems in-troduces a new security dimension. Specifically, memory is retrieved during task execution and directly shapes tool-use decisions. ## 2. Related Work 

Control Flow Attacks in Agents. Modern agents rely on external tools (Schick et al., 2023), making their be-havior governed by the tool -call trace. Accordingly, con-trol flow attacks (Li et al., 2026) aim to manipulate this trace, steering tool selection and invocation sequences to in-duce unintended actions. For example, attractive-metadata attacks bias tool choice by manipulating tool descriptors (Mo et al., 2025); obfuscated prompt attacks induce tool-connected agents to leak sensitive data(Fu et al., 2024); Tool-mediated prompt injection hijacks tool use via un-trusted tool outputs or tool-ingested content (Zhang et al., 2025a). Security analyses further systematize these threats: Agent Security Bench catalogs tool -based attack surfaces, including indirect injections within tool responses (Zhang et al., 2025b), while AGENT - SAFETYBENCH benchmarks behavior - level safety risks introduced by tool use (Zhang et al., 2024). These threats can also be multimodal. For example, adversarial images can trigger harmful tool calls (Fu et al., 2023); INJECAGENT shows that indirect injec-tions in tool - returned content can drive attacker -intended tool calls (Zhan et al., 2024); and third - party API response attacks demonstrate how subtle output manipulation can poison downstream agent behavior (Zhao et al., 2024). 

Memory Poisoning and Long-Term Influence. Agent memory is increasingly central to planning and long -horizon behavior, making it a natural target for poisoning attacks (Park et al., 2023). Prior work has explored several ways adversaries can corrupt or exploit stored information. Agent-Poison injects backdoors into agent knowledge bases (Chen et al., 2024), while MINJA shows that interaction -driven injections can become self - reinforcing as corrupted prece-dents accumulate over time (Dong et al., 2025). BadChain 2From Storage to Steering: Memory Control Flow Attacks on LLM Agents 

demonstrates that poisoning a small set of reasoning exem-plars can create triggerable prompt -only backdoors (Xiang et al., 2024). Beyond behavioral manipulation, MEXTRA reveals that memory modules can leak private user–agent interaction data when adversarially probed (Wang et al., 2025a). Defensive efforts in this domain mainly focus on detecting or isolating corrupted memory. A-MemGuard performs online consistency checks and separates “lesson” memory to mitigate poisoning (Wei et al., 2025). Llama Guard filters risky prompts and responses using a predefined safety taxonomy (Inan et al., 2023), and ISOLATEGPT pro-poses execution isolation to limit cross -component influence in LLM application ecosystems (Wu et al., 2025). Taken together, prior analyses of control -flow attacks treat each task as an isolated, one -off session, overlooking the per-sistent influence that memory can exert in agentic systems. Conversely, existing memory -attack studies focus primarily on degradation in retrieved content or output correctness, overlooking its substantial influence on an agent’s control flow. This gap motivates our study: we treat memory as a control input to evaluate persistent biases in tool selection and ordering via the tool-call trace. In modern agents, poi-soned memory can act as a write-once, read-many control signal, inducing cross-task control flow deviations. 

## 3. Formalization of Memory Control Flow Attacks and Experimental Verification 

This section formalizes the Memory Control Flow Attacks (MCFA) with LLM agent as a state machine driven by mem-ory retrieval. Based on that, we propose a Verification The-orem that establishes the sufficiency of trace-based auditing under context isolation. Finally, we formalize a experimen-tal methodology to validate these theoretical bounds. 

3.1. Memory Control Flow Attacks Security Goals & Assumptions . We consider a memory-augmented LLM agent with access to tools, where retrieved memory is injected into the AgentRunner, and tool-call traces serve as an observable proxy for the agent’s control flow. The adversary is an external attacker, who through one or a few standard interactions, induces the agent to store action-oriented preferences/rules into long-term memory. During later benign tasks, these rules may be retrieved and our security objective is control flow integrity rather than output correctness; an attack succeeds if retrieved malicious memory causes auditable policy deviations in the tool trace. During the interactions, we assume no internal privileges in the interactions, i.e. the attacker cannot modify system prompts, alter tool implementations, access the memory store directly, or implant malicious tools. As a production-style mitigation to reduce MCFA, we consider Role-Based Memory Segregation (RBMS), which separates system rules from user preferences and enforces an explicit priority hier-archy between them. Under RBMS, the attacker can only affect the user-memory channel. Additional details are pro-vided in Appendix A. Under this security objective and assumption, we define the system model and memory con-trol flow attack (MCFA) as follows. 

Definition 1 (System Model: Memory-Augmented LLM Agents) . Given F is the stochastic transition function of the LLM, mapping context to actions and T the discrete set of available tools {t1, . . . , t N }, we formally define a memory-augmented agent A as a tuple:        

> A=⟨F, T, M, Φret ,Φupd ⟩,(1)

Φret (Mk, x ) the retrieval function generating context from memory given input x and Φupd (Mk, δ ) is the update func-tion Mk+1 ← Φupd (Mk, δ ).

Definition 2 (Memory Control Flow Attack) . An MCFA is a two-phase process where an adversary injects a policy 

πadv into M such that for a subsequent benign task x, the generated tool trace τ satisfies τ |= πadv and τ̸ |= Π saf e ,where Πsaf e is the system’s default safety policy. 

3.2. Verification: Memory-Causal Deviation of MCFA 

To rigorously detect MCFA, we must distinguish it from short-term context hallucinations. We propose the follow-ing theorem which serves as the theoretical basis for our evaluation framework M EM FLOW .

Theorem 1 (Memory-Causal Verification) . Let H be the short-term conversation history and Eiso be an Isolated Evaluation Regime where H = ∅. For any agent A and task x, if the tool trace τ generated under Eiso violates the safety policy Πsaf e (i.e., Dev (τ, Πsaf e ) > 0), then the deviation is strictly attributable to the memory state M .Proof. The agent’s action generation is a function P (τ |

x, H, Φret (M )) . Under the Isolated Regime Eiso , we force 

H → ∅ . The function simplifies to P (τ | x, Φret (M )) .Given that x is benign (contains no adversarial prompts) and the base model F is aligned to Πsaf e (by assumption of the victim model), the only variable capable of introducing a sys-tematic policy shift πadv is the retrieved context Φret (M ).Therefore, Dev (τ ) > 0 = ⇒ M is poisoned. 

Remark 1. This theorem provides the necessary condition for attribution. In practical security auditing, distinguishing a model hallucination (stochastic noise) from a persistent attack is challenging. 3.3. Methodology: M EM FLOW 

Trace Space and Attack Mappings . Let Arg denote the universe of valid argument structures (e.g., JSON objects or parameter lists) accepted by the toolset T . We define the Trace Space Ωτ as the set of all finite sequences of pairs of (a selected tool, instantiated arguments): 

Ωτ = {τ = (( t1, α 1), . . . , (tL, α L)) | ti ∈ T , α i ∈ A rg } ,

(2) 

where at each step, the agent generates a selected tool ti and its instantiated arguments αi.Through a variety of preliminary experiments, we find that MCFA can induce control flow deviations in multiple ways and exhibit diverse characteristics. To systematically cap-ture these behaviors, we formalize five attack families as distinct mapping functions f : Π adv → P (Ω τ ), mapping an adversarial instruction πadv to a target subspace of Ωτ .The families, along with their definitions and interpretations, are summarized in Table 1. O VERRIDE and O RDER capture two forms of control - flow deviations: tool - choice overrides and reordered workflows. In contrast, M -SCOPE , P ERSIST ,and R ELAPSE characterize whether a deviation, once ob-served, propagates across tasks (cross - task propagation), how long it endures (long -horizon persistence), and how resistant it is to mitigation (mitigation resistance). Because cross - task propagation, long -horizon persistence, and miti-gation resistance arise from a stable, task - agnostic control policy encoded in memory, we instantiate these families using tool - choice override as the deviation behavior. See Appendix B for concrete examples. 

The Trace Auditing Algorithm . Given a specific attack family, we define attack success as the trace τ falling into its corresponding violation sets in the Trace Space Ωτ . To further evaluate the attack success, we use the attack suc-cess rate (ASR) as the metric. Based on Theorem 1, we construct the isolation-based trace auditing algorithm in Algorithm 1, which audits memory-induced control flow deviations through a two-phase protocol: injection and iso-lated evaluation. Given an agent A, initial memory Minit , an injection prompt Pinj , and a benign task set Xbenign , Phase 1 performs a one-shot update that writes the injected content into memory, yielding M ′ ← Φupd (Minit , P inj ).In Phase 2, for each benign task xi ∈ Xbenign , we enforce an isolation regime by resetting the short-term history to 

H ← ∅ (Theorem 1), ensuring that any observed deviation is attributable to retrieved long-term memory rather than residual dialogue state. We then retrieve context from the poisoned memory, Rctx ← Φret (M ′, x i), and execute the agent to obtain a tool-call trace τi ← F (xi, R ctx ). We flag a deviation if the resulting trace violates the default safe control flow policy, i.e., τi /∈ Πsafe , and accumulate the count S. MEM FLOW Evaluation Framework . To systematically quantify the threat boundaries and efficacy of memory con-trol flow attacks (MCFA), we developed M EM FLOW , the first automated evaluation framework designed for agents equipped with memory. Unlike traditional evaluations that focus on text generation quality, M EM FLOW focuses on auditing the control flow integrity . Through standardized 

Injection-Auditing by Algorithm 1, M EM FLOW assesses whether an agent’s tool traces systematically deviate from default safety policies after injected untrusted memory. The system architecture of M EM FLOW , as shown in Figure 2, consists of three core components: 

❶ Scenario Generation. To decouple testing logic from the underlying frameworks, we introduce a unified Scenario 

abstraction layer. Each evaluation scenario is formalized by a set of specifications: • Tool Specs : Defines the tool pool sourced from real world, including the safe tools ( Tsafe ), the correspond-ing risky variants ( Trisky ) constructed from the same base tools, and the memory write interfaces. • Policy Constraints : Defines the expected legitimate control flow. • Dynamic Injection : Utilizes an LLM to dynamically generate diverse natural language instructions, simu-lating authentic User-Memory interactions rather than relying on hard-coded prompt attacks. 

❷ AgentRunner . To evaluate the universality of attacks, we build the AgentRunner to provide a unified interaction interface across mainstream frameworks such as LangChain and LlamaIndex, supporting LLMs (OpenAI, Anthropic, Gemini) with pluggable memory components. This design ensures strict comparability of security performance across different agent architectures under the same test data. 

❸ Trace-audited Pipeline. MEM FLOW standardizes each evaluation trial and calculates the ASR via Algrithm 1. Through this automated workflow, M EM FLOW enables rapid identification of security vulnerabilities in different memory mechanisms under batch execution modes.

resulting in Attack Success Rates (ASR) approaching 1.0

under strong retrieval. Remark 2. This corollary implies a fundamental alignment bypass threat. LLMs are supposed to prioritize system prompts over instructions. However, the RAG mechanism 5From Storage to Steering: Memory Control Flow Attacks on LLM Agents 

can inject the adversarial instruction into the context window as “ground truth” or “user preference”. In this circumstance, LLM tends to faithfully execute the retrieved adversarial instruction that facilitates the attack, effectively bypassing the static safety constraints encoded in the system prompts. We provide a detailed empirical analysis of this phenomenon in §4.2 (RQ1 & RQ2). 

Corollary 2 (Inefficacy of Textual Repair) . Let Rep be a natural language repair instruction. The persistence prop-erty implies: 

P (τsaf e | Mpoisoned , Rep ) ≈ 0,

indicating that removing πadv from M is necessary; mask-ing it with Rep is insufficient. Remark 3. This highlights the limitation of stateless de-fense . Most current defenses treat the agent as stateless, attempting to fix behavior through conversational prompts alone. However, since the state M remains poisoned, the retrieval function Φret will re-introduce πadv in future turns. Effective mitigation therefore requires Memory Surgery (di-rect state modification) rather than conversational steering. We provide a detailed empirical analysis of this phenomenon in §4.2 (RQ3). 

## 4. Experiments 

4.1. Experimental Setup and Implementation LLM backbones . We evaluate all experiments on gpt-5mini , Claude Sonnet 4.5 , Gemini 2.5 Flash under the same scenario specifications and tool inventories for fair compari-son.We set temperature to 0.0; therefore each configuration is executed under a fixed seed policy. We also conduct experiments on open-source LLM, which are detailed in Appendix E. 

Tools . We sample 36 tool candidates per framework from 169 LangChain community tools (LangChain, 2026) and 97 LlamaHub tools (LlamaIndex, 2026), using stratified random sampling to balance tool categories. Detailed in-formation regarding the LangChain community tools and LlamaHub tools is provided in Appendix F. We also use Tool target to indicate which tool the system prompt prioritizes: safe means the agent is instructed to prefer the tool explicitly marked as SAFE, while none 

means no tool is designated as preferred in the system prompt. Tool label is applied to show how the malicious tool is described in its own metadata: harmful means the tool is explicitly labeled as harmful, whereas neutral means it is presented as neutral. Examples of Tool label can refer to Appendix D. 

Memory Retrieval. In all experiments, memory is con-structed by storing the key information i.e. user preference, from session conversations and can be retrieved in three ways: Strong , which strictly enforces the system prompt directive to “follow user preference”; Weak , which en-forces this directive only loosely; and OFF , which disables long - term retrieval entirely. We present detailed system prompt examples in Appendix C. 

Interaction Regime. An Isolated regime means that, after each session ends, all in-session context is cleared and the 

AgentRunner is rebuilt. By contrast, an Online regime keeps all conversations within a single session without re-building the runner. We aim to use this discrimination to clarify whether the observed deviations arise from retrieved memory or from accumulated historical interactions.

4.3. Ablations Study

## 5. Conclusion 

This paper presents the first systematic security analysis of the impact of memory on the control flows of LLM agents. We identify a novel threat, Memory Control Flow Attack (MCFA), which exploits memory read/write chan-nels to covertly hijack tool selection and disrupt workflow execution order across tasks. Building on this foundation, we present M EM FLOW , an evaluation framework that stan-dardizes injection and auditing and measures controlflow integrity via tool-call traces. Across mainstream tool frame-works and state-of-the-art LLM backends, we find that MCFA can effectively yield pervasive control flow hijack-ing. We further evaluate a production-style mitigation, dual-channel memory with role-based segregation (RBMS): it reduces attack success but remains imperfect due to hierar-chy non-compliance under strongly phrased, action-oriented memories. These findings underscore the critical need for rigorous memory auditing mechanisms and isolation archi-tectures in future agent systems. 

## Impact Statement 

This paper aims to advance the security of LLM-based agents by exposing structural vulnerabilities in memory mechanisms. We recognize that the proposed Memory Con-trol Flow Attack (MCFA) could potentially be misused by malicious actors. However, we believe that systematically identifying and quantifying this threat surface is a prereq-uisite for developing robust defenses, such as memory iso-lation and auditing mechanisms. By releasing M EM FLOW ,we empower developers to assess and mitigate these risks prior to real-world deployment, ultimately contributing to the safe and responsible adoption of autonomous agents.

## Appendix A. Threat Model 

This section characterizes the adversarial setting studied in this paper. The adversary requires neither the ability to forge or implant malicious tools, nor the need to breach system privilege boundaries. Instead, through one or a few turns of standard interaction, the adversary induces the agent to write “action-oriented” instructions or preferences into its long-term memory . These corrupted memories are subsequently retrieved during normal tasks, causing systematic deviations in tool selection and control flow . We term this threat Memory Control Flow Attacks (MCFA) .

A.1. Victim System & Trust Boundary Victim System: We consider the victim system to be a tool-augmented LLM Agent equipped with long-term memory, consisting of the following components: • AgentRunner: Generates the next action plan based on the current user query, dialogue context, available tool descriptions, and, critically, retrieved long-term memory . The plan includes which tools to select, the order of invocation, and how to process tool outputs. • Tool Set: A set of tools pre-configured by developers. The system typically adheres to a “default safety policy,” such as prioritizing safe tools or following a specific invocation order. • Persistent Memory: Information preserved across turns and tasks. Memory is retrieved at the beginning of a task or during the planning phase and is written back in the form of “summaries/preference updates” at the end of a task. • Memory Architecture (Naive vs. RBMS): We consider two common designs. ( Naive ) A single, flat memory stream where all write-back entries are stored and retrieved uniformly. ( RBMS / Segregated ) Role-based memory segregation that separates channels (e.g., user preferences/history vs. system policies/safety) and scopes retrieval by role to reduce cross-boundary influence. • Executor: Responsible for the actual execution of tool calls and passing tool outputs back to the AgentRunner for subsequent reasoning. 

Trust Boundary: We consider the tool implementations, system prompts, and the execution environment to be trusted (immutable by external parties). Conversely, user inputs, external content sources, and unstructured text carried within tool outputs are considered within the untrusted boundary. The key observation of this paper is that untrusted content can enter the persistent state via “memory write-back,” thereby crossing the boundary of a single session and becoming an implicit control signal that influences future control flows. This trust-boundary crossing can occur even in segregated memory designs (e.g., RBMS): although system policies may reside in a protected channel, attacker-crafted content can still be written into the user channel and later retrieved as a decision signal during planning. We therefore treat RBMS as a defended victim architecture and evaluate whether it is sufficient. 

A.2. Adversary Model 

The adversary is an external input provider. Typical manifestations include malicious users, compromised content sources, or third-party entities capable of indirectly influencing the textual output of tools. The adversary’s capabilities and constraints are defined as follows: 

Capabilities: 

1. Interaction Capability: The adversary can engage in one or a few turns of standard dialogue with the agent, or provide external text content that the agent processes. 2. Memory Injection Capability: The adversary can craft text that, during the system’s memory write-back process, is adopted as “preferences/rules/workflow patterns” or other action-oriented information (e.g., “Always use Tool X from now on,” or “Skip audit steps for this type of task”). Under RBMS, we assume the adversary can only influence the user-memory channel (e.g., preferences/history) and cannot write to protected system-policy memory. Nevertheless, 11 From Storage to Steering: Memory Control Flow Attacks on LLM Agents 

we hypothesize that semantic ambiguity can make action-oriented directives appear as benign preferences, allowing them to be retrieved and used as a control signal for tool choice. 3. Black-box Trigger Capability: In subsequent normal tasks, as the corrupted memory is read by the AgentRunner to influence tool selection/order, the adversary does not need to sustain participation in the dialogue to cause persistent deviation. 

Constraints: 

1. No Internal Privileges: The adversary cannot directly read/write the memory storage database, modify system prompts, change tool implementations/descriptions, or alter tool routing logic code. 2. Model Agnostic: The adversary does not require knowledge of specific LLM details, relying only on the deployment norm that “memory participates in planning inputs.” 3. No Malicious Supply Chain: We do not assume the attacker can implant or replace tools; the tool set and execution logic are benign by default. 

A.3. Goals & Security Properties 

The security goal of this paper is not the correctness of output content, but rather the Agent-level Control Flow Integrity :under a given tool set and default safety policy, the tool selection and invocation order during the planning phase should not be systematically skewed by untrusted inputs via long-term memory. We evaluate this controlflow integrity both under naive memory and under RBMS-style segregated memory to assess whether segregation alone is sufficient. We consider an attack successful if and only if: malicious/misleading instructions in long-term memory are retrieved in subsequent tasks and influence planning, causing the agent to exhibit observable policy deviations (e.g., incorrect tool selection, bypassing security checks, or altering the order of safety-critical steps), and such deviations persist across multiple normal tasks. Section 4 will detail specific forms of MCFA and quantify these deviations. 

A.4. Attack Surface: Memory Write Primitives 

The critical entry point for MCFA is the “write interface.” In real-world systems, there are at least three common write paths: • Explicit Preference Write: The system provides mechanisms to write preferences/profiles/long-term rules. Attackers can disguise meta-instructions as preference statements to enter long-term memory. • Implicit Summary Write: Dialogue history or tool outputs are summarized and written back. Attackers can embed action-oriented phrasing in dialogue or external text, inducing the summarizer to solidify them into “rules.” • Retrieval Cache Write: RAG caches, search indices, or “recent usage patterns” are written back. Attackers pollute content or ranking signals, causing erroneous/harmful “evidence” to persist and continuously influence planning. Our threat model emphasizes that once any of the above paths allow untrusted content to be written and subsequently read during planning, an amplification chain of “ Write Once → Read Many → Stable Control Flow Deviation ” is formed. 

A.5. Success Criteria 

To make the threat model operational and measurable, we use observable control flow evidence as success criteria, rather than relying on the model’s self-explanation. Specifically, we focus on the following auditable events: • Whether a systematic shift in tool selection occurs. • Whether changes occur in tool invocation order and dependency relationships. • The coverage scope and duration of the deviation across a sequence of tasks. • Whether the deviation relapses (resistance) after correction/intervention attempts. These criteria correspond directly to the metric designs in our subsequent benchmarks (Override, Order, M-Scope, Persistence, Resistance). 12 From Storage to Steering: Memory Control Flow Attacks on LLM Agents 

A.6. Out-of-Scope 

To focus on the distinct threat surface of “Memory Control Flow Attack,” the following scenarios are considered out of scope: • Malicious Tool Supply Chain / Tool Impersonation: Injecting malicious tools to steal/tamper with data during execution. • Traditional Prompt Injection: Attacks that affect only a single turn’s output without entering long-term memory or causing cross-task control flow deviation. • System-level Compromise: Direct tampering with system prompts, tool implementations, execution environments, or direct read/write access to the memory database. • Side-channel and Physical Attacks: Attack surfaces unrelated to the mechanism discussed in this paper.

## E. Tool Hubs and External Integrations 

Modern agent frameworks rarely ship all tools in the core package. Instead, they rely on tool hubs , where each integration is distributed as an external package. This ecosystem design significantly expands the tool surface available to agents, but also increases the attack surface when memory can steer tool choice and workflow execution. 

E.1. LlamaIndex Tool Hub: LlamaHub LlamaIndex: LlamaHub. LlamaHub is LlamaIndex’s integration hub that provides reusable components including data loaders and agent tools. It also offers Tool Specs , i.e., bundles of related tools for a single service/interface that can be installed as external packages and registered to an agent. 

E.2. LangChain Tool Ecosystem: Tools via Integrations Packages 

LangChain exposes tools and toolkits via an extensive integration ecosystem, where many connectors are distributed as standalone packages (including community-maintained integrations). These external packages allow developers to add third-party tools/toolkits to an agent without modifying the core framework.
