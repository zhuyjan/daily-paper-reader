## 摘要
训练工具增强的大语言模型代理需要大量多轮、基于工具对话数据。这些数据标注成本高昂，在生产环境中受隐私限制，且公共数据集中基本缺失。我们提出StateGen，一个合成数据生成平台，通过编排四角色LLM循环生成带有评分和丰富推理轨迹的训练对话：角色条件化的用户模拟器、受测代理、基于状态的工具模拟器以及多轴LLM评判器。关键的架构贡献是一个权威状态管理器，它在多轮对话中维护一个结构化的世界状态对象，强制执行“后端即真相”的不变性，从而在构造上消除了最主要的工具调用幻觉类型。StateGen通过将子代理声明为工具，并共享单一状态对象，自然地扩展到层次化多代理设置。我们在三个生产语料库上报告了64,698个评估对话的结果：工具调用幻觉得分达到9.66/10，系统通过23维特征向量支持角色驱动的变化，并且清晰分离的训练集和黄金评估集划分证实了数据并非记忆诱饵（每准则差距分析）。与八个外部系统的比较表明，没有单一公开可用的平台能同时结合多轮生成、基于状态的工具模拟、层次化多代理支持以及内置评判器评分。

## Abstract
Training tool-augmented LLM agents requires large corpora of multi-turn, tool-grounded conversational data that is expensive to annotate, privacy-constrained in production settings, and largely absent from public datasets. We present StateGen, a synthetic data generation platform that produces scored, reasoning-trace-rich training conversations by orchestrating a four-role LLM loop: a persona-conditioned user simulator, an agent under test, a state-grounded tool simulator, and a multi-axis LLM judge. The key architectural contribution is an authoritative state manager that maintains a structured world-state object across turns, enforcing a backend-is-truth invariant that eliminates the dominant class of tool-call hallucinations by construction. StateGen extends naturally to hierarchical multi-agent settings by declaring sub-agents as tools, all sharing a single state object. We report results on 64,698 evaluated conversations across three production corpora: tool-call hallucination scores reach 9.66/10, the system supports persona-driven variation via a 23-dimensional trait vector, and a cleanly separated train and golden evaluation set split confirms the data is not memorization bait (per-criterion gap analysis). Comparison with eight external systems shows that no single publicly available platform combines multi-turn generation, state-grounded tool simulation, hierarchical multi-agent support, and built-in judge scoring.

## 1. 核心问题与整体含义（研究动机与背景）
- **数据瓶颈**：训练工具增强的大语言模型（LLM）代理需要大量多轮、工具接地的对话数据。人工标注成本极高（每人每小时仅产出10–40条，多数不合格，10K样本需数月）；生产日志包含PII等敏感信息，难以直接使用；公共数据集（如ShareGPT、Dolly）几乎不含工具使用轨迹，评估基准（τ-bench、AgentBench）仅提供数百条评估用轨迹，无法满足微调所需数万条规模。
- **目标**：构建一个合成数据生成平台，填补训练规模与评估规模之间的数据鸿沟，同时解决工具调用幻觉、人物多样性、层次化多代理支持和自动评分等关键需求。

## 2. 方法论：核心思想、技术细节与流程
### 2.1 核心思想
- **四角色LLM循环**：用户模拟器、被测代理（Agent under Test）、工具模拟器、多轴LLM评判器协同工作，生成带评分和推理轨迹的对话。
- **权威状态管理器**：维护结构化的世界状态对象 \( S_t \)，强制**“后端即真相”** 不变性——事实进入状态仅当来自于初始状态或合法的工具写操作，从而从构造上消除工具调用幻觉。
- **层次化多代理**：将子代理声明为工具，所有子代理共享单一状态对象，通过能力模板实现零代码子代理创作。

### 2.2 关键技术细节
#### 输入规格
- **场景**（YAML）：用户事实 \( F_u \)、系统事实 \( F_s \)、用户目标 \( g \)、期望最终状态 \( S^* \) 等。
- **人物向量**（23维）：6个分类人口属性 + 12个连续行为特征 + 5个连续情绪状态（随场景变化）。
- **剧本**：编排器系统提示、工具定义与JSON Schema、工具能力模板、可选护栏库。

#### 角色温度设置
| 角色 | 温度 | 理由 |
|------|------|------|
| 用户模拟器 | 0.7 | 自然语言变化 |
| 被测代理 | 0.7 | 匹配生产推理 |
| 工具模拟器 | 0.3 | 状态一致性优先 |
| 状态管理器 | 0.0 | 确定性状态更新 |
| LLM评判器 | 0.2 | 轻微随机用于打破平局 |

#### 生成流程（算法1）
1. 初始化状态 \( S \leftarrow F_s \)。
2. 循环最多10轮（每轮最多5次工具调用）：
   - 用户模拟器生成查询 \( q_t \)。
   - 代理生成工具调用 \( a_{st} \) 或终止动作。
   - 非终止时：工具模拟器基于当前状态 \( S \) 和工具能力模板生成响应 \( r_{st} \)。
   - 状态管理器更新 \( S \)。
3. 对话结束，评判器给出8轴评分向量 \( \mathbf{j} \in \{1,\dots,10\}^8 \)。

#### 状态管理器的不变性
- 事实进入 \( S \) 当且仅当：(a)来自初始 \( F_s \)，或(b)由合法工具调用写权限创建。系统提示明确要求：“如果状态说事务存在，必须找到；如果说不存在，必须返回对应错误”。

#### 人物模型
- **三层人物向量**：人口属性（独立采样）、行为特征（围绕基向量加高斯噪声 \( \sigma=0.08 \)，分低/中/高三档）、情绪状态（从区间随机初始化+场景依赖偏移，如纠纷场景增加挫败感+0.25，降低信任-0.20）。
- **特征相关**：通过编码基向量和七条规则描述实现，无全协方差建模。
- **查询复杂度叠加**：简单/中等/复杂/模糊四层。

#### 8轴LLM评判器
默认八条准则：目标达成、工具使用、工具调用幻觉、推理质量、推理幻觉、沟通质量、一致性、错误处理。每个项目可增删或加权重。评分基于完整对话和期望最终状态 \( S^* \)，LLM自行决定总分（非确定性聚合）。

#### 推理轨迹捕获与导出
- 支持 reasoning_content 或 <think> 标签等格式，输出为 JSONL（每行含 messages, tools, metadata），评分信息作为元数据用于过滤性蒸馏。

#### 场景验证流水线
- 第一阶段：程序化检查（JSON Schema、引用完整性、重复检测，相似度阈值0.85/0.90）。

## 7. 优点
- **状态接地机制**：权威状态管理器从根本上消除幻觉传播，比纯提示式工具模拟更可靠。
- **层次化多代理生成**：通过子代理即工具模式，无需手写仿真代码即可为编排-子代理系统生成训练数据。
- **丰富的人物模型**：23维向量+查询复杂度叠加提供了可测量的行为变化，避免重复样本。
- **多轴评判器**：8个独立评分+扩展性，支持按失败模式过滤训练数据（如单独强调工具幻觉）。
- **推理轨迹捕获**：支持蒸馏大型推理模型到小型模型，生成JSONL可直接用于微调。
- **场景验证流水线**：程序化+LLM语义检查双重保障，避免无效场景产生坏数据。
- **训练/评估分离**：零场景重叠的黄金集，能测试模型泛化而非记忆。

## 8. 不足与局限
- **评判器未校准**：绝对分数可能存在评判模型自身的系统偏差，缺乏人类黄金标注验证。计划收集1-2K人工样本进行偏差校正。
- **人物协方差模型简单**：目前依赖规则描述而非从数据学习，缺乏真实世界的共现模式。作者计划从64K语料库拟合协方差矩阵来弥补。
- **总体不透明**：总分为LLM自由选择而非确定性函数，造成可解释性不足。需未来明确聚合规则或记录经验关系。
- **状态管理器仍是LLM**：尽管温度0，复杂状态更新（多记录事务、部分回滚）可能引入细微不一致。正在研究混合方法（结构化操作用确定性逻辑）。
- **缺少消融实验**：未量化每个组件（状态管理器、人物向量各层、评判器各轴）的独立贡献。
- **计算成本未报告**：缺乏生成64K对话的GPU时间、成本数据，影响了实用性评估。
- **未处理工具故障注入**：当前工具模拟只返回正常响应，未模拟超时、限速、格式错误等故障情形，限制了训练代理的优雅降级能力。作者列为近期扩展计划。
- **仅与少数外部系统对比**：未与更多商用合成数据平台（如MOSTLY AI）进行详细功能和性能对比，且对比基于文档而非实验复现。
