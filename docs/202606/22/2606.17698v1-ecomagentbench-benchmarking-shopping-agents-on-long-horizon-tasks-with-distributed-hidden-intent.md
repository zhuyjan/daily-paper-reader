## 摘要
随着基于大语言模型的购物代理进入生产环境，现有基准测试未能捕捉购物者需求的出现方式：这些需求可能隐含在查询中陈述、记录在个人档案中，或者仅在提出正确问题时才显现。那些提前暴露全部意图且仅评价最终选择的基准测试既无法提出这种长周期挑战，也无法解释代理遗漏了哪项需求。为填补这一空白，我们提出了EComAgentBench，这是一个基于真实亚马逊商品和评论的662个任务基准测试。每个任务将这些需求分散到可见查询、工具限制的个人档案以及脚本化澄清中；代理必须揭示隐藏意图，根据属性和评论证据验证候选商品，并在100次工具调用内确定唯一商品。此外，带有类型和来源标签的评分细则对每个任务进行评分，将每次失败归因于某条需求及其来源。构建过程自动化且可靠，每个答案在生成任何文本之前就已固定于代码中，每个样本均经过验证。我们对七个模型的评估显示，即使是最强模型也只能达到57.1%的整体准确率，并且评分细则的满足度从可见来源到隐藏来源逐渐下降。总体而言，我们相信EComAgentBench将成为将购物代理从单查询搜索推向长周期可靠辅助的可复现基础。

## Abstract
As LLM-based shopping agents enter production, existing benchmarks fail to capture how a shopper's requirements arrive: stated implicitly in the query, recorded in a profile, or revealed only when the right question is asked. Benchmarks that expose full intent upfront and grade only the final choice can neither pose this long-horizon challenge nor explain which requirement an agent missed. To address this gap, we introduce EComAgentBench, a benchmark of 662 tasks grounded in real Amazon products and reviews. Each task scatters these requirements across a visible query, a tool-gated profile, and scripted clarification; an agent must uncover hidden intent, verify candidates against attributes and review evidence, and commit to a single product within 100 tool calls. Moreover, typed, source-tagged rubrics grade every task, attributing each failure to a requirement and its source. Construction is automated yet reliable, with every answer fixed in code before any text is generated and every sample validated. Our evaluation of seven models reveals that even the strongest attains only 57.1% overall accuracy, and rubric satisfaction degrades from visible to hidden sources. Overall, we believe EComAgentBench will serve as a reproducible foundation for moving shopping agents from single-query search toward dependable assistance over long horizons.

- **核心问题**：现有基于大语言模型的购物代理基准测试存在三大缺陷：

  - **构建不可靠**：长周期任务人工标注成本高且裁判噪声大，多数基准缺乏验证、自动化或真实产品支持。
- **整体含义**：为了推动购物代理从单查询搜索向长周期可靠辅助演进，需要一种能够模拟需求分布隐藏、提供可诊断失败原因的基准测试。EComAgentBench 正是为此设计，通过真实产品目录和自动化构建，为研究社区提供可复现的评估基础。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：将购物任务的需求分散到三个来源，代理必须通过工具调用逐步恢复完整意图，并最终推荐一个产品。
- **技术细节**：
  - **三个来源**：可见查询（部分需求仅隐含陈述）、工具门控档案（需调用 `get_user_profile` 获取）、脚本化澄清（需通过 `ask_user` 触发固定答复）。
  - **环境设置**：一个 API 沙箱，包含 10 个工具（搜索、过滤、产品详情、评论证据、用户交互、计算、推荐等），每轮上限 100 个工具步和 10 次澄清轮次。
  - **评分规则（rubrics）**：每个规则带有 **类型**（`attribute_match`, `numeric_range`, `entity_match`, `negative_attribute`, `budget_match`, `review_opinion`）和 **来源标签**（query, persona, clarification）。答案在代码中预先固定，LLM 仅负责语言表述，不影响正确性标准。评分采用双路径：精确产品匹配或全部规则满足。
  - **自动化构建流程**：七阶段流水线（采样、特征提取→分配来源→编译规则→语言生成→分层验证→有效拆分），每个样本经过自动检查（防泄漏、档案安全白名单、澄清完整性、语义验证）。验证通过率 71.9%（921 样本中 662 有效）。
  - **裁判审计**：使用 `gemini-3.1-pro-preview` 作为统一裁判，审计显示最多每个模型 1 次错误拒绝，且无错误接受（对 GPT-5.4 的 161 条规则全部再验证为正确）。

## 6. 论文的主要结论与发现

- **模型分离能力**：准确率跨度 19.5%–57.1%（Opus 最高），区分度良好。
- **隐藏意图更难**：所有模型在查询源的规则满足度（68.8%–88.1%）显著高于档案（43.1%–72.8%）和澄清（45.4%–70.9%）。
- **隐式需求密度是关键难度驱动**：隐式需求数量从 0–1 增至 3+ 时，平均准确率从 51.8% 骤降至 27.2%；而总规则数量影响不大。
- **失败集中在评论证据**：`review_opinion` 类型规则满足度最低（33.7%–67.4%），是比结构化属性匹配更难的能力。
- **两种交互风格**：深度搜索/晚提问（Opus、Kimi、MiniMax）与浅搜索/早提问（GPT 系列），均未完全解决隐藏需求，且深度搜索更易耗尽步数预算（Opus 完成率 84.6%）。
- **裁判噪声极小**：错误拒绝≤1 次/模型，无错误接受，验证了难度来自任务本身。

## 7. 优点：方法或实验设计上的亮点

- **创新性任务设计**：将需求分布在三个工具门控来源，构造了真正的长周期隐藏意图发现挑战，更贴近真实购物场景。
- **可诊断评分**：类型和来源标签化的规则使失败可归因，能定位代理在哪个需求以及来自哪个来源出错，优于现有粗粒度评分。
- **自动化且可靠的构建**：答案在代码中固定，LLM 仅负责语言表述，消除了正确性标准的变异；多阶段验证确保样本质量；构建管道是类别无关的，可扩展。
- **裁判可靠性验证细致**：利用双路径正确性定义进行双向审计，证明裁判几乎无错误，增强了结果的可信度。
- **丰富的分析维度**：从准确率、规则满足度、隐式密度、交互行为、意图类型、失败类型等多个角度剖析，提供了深入见解。

## 8. 不足与局限

- **场景覆盖有限**：产品集中于电子相关类别；所有任务为英文、单一亚马逊市场；每个任务仅推荐单个产品，未涉及多物品购物车、比较对话、售后等。
- **用户侧脚本化**：档案是合成的声明偏好（可能有模型生成偏见），澄清完全确定，牺牲了自然“语音”以换取归因可重复性；未尝试从行为信号推断隐藏偏好。
- **裁判依赖与残留噪声**：尽管审计显示噪声极小，但无法完全排除 LLM 裁判的残留偏差；裁判 `gemini-3.1-pro-preview` 的选择本身可能引入偏好。
- **缺乏消融实验**：未定量分析不同来源复杂度对准确率的具体贡献比例，例如移除澄清后性能变化。
- **伦理考量**：档案可能继承生成模型的人口统计偏见；产品目录版权问题（仅引用原始数据集）。
