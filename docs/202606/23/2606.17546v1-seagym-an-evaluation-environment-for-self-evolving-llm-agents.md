## 摘要
自演化基于大语言模型的智能体主要通过改变其智能体框架（即围绕基础模型的结构化执行层，包括提示、记忆、工具、中间件、运行时状态以及模型-工具交互循环）来提升性能。现有评估往往将此过程简化为孤立的任务分数或单一序列曲线，难以判断更新是否产生了可复用的改进、是否过拟合近期任务、是否增加了成本或损害了旧有行为。我们提出SEAGym，一个用于评估智能体框架更新在训练、验证、测试、重放和成本记录上的评估环境。SEAGym将兼容Harbor的基准测试转化为动态自演化任务源，包含训练批次、冻结更新验证、保留的域内和域外迁移视图、重放诊断以及保存的快照和指标记录。在Terminal-Bench 2.0和HLE上实例化SEAGym，我们在共享的周期/批次协议下比较了ACE、TF-GRPO和AHE。结果表明，这些评估视图提供了关于演化过程的互补信号：频繁更新可能无法提升保留性能，有用的中间快照可能随后崩溃，源多样性和模型后端可能影响框架可靠性。

## Abstract
Self-evolving LLM-based agents improve mainly by changing their agent harness: the structured execution layer around a base model, including prompts, memory, tools, middleware, runtime state, and the model-tool interaction loop. Existing evaluations often reduce this process to isolated task scores or a single sequential curve, obscuring whether an update produces reusable improvement, overfits recent tasks, increases cost, or harms older behavior. We introduce SEAGym, an evaluation environment for measuring agent harness updates across training, validation, test, replay, and cost records. SEAGym turns Harbor-compatible benchmarks into dynamic self-evolution task sources with train batches, frozen update-validation, held-out ID and OOD transfer views, replay diagnostics, and saved snapshot and metric records. Instantiating SEAGym on Terminal-Bench 2.0 and HLE, we compare ACE, TF-GRPO, and AHE under a shared epoch/batch protocol. The results show that these evaluation views provide complementary signals about the evolution process: frequent updates may fail to improve held-out performance, useful intermediate snapshots may collapse later, and source diversity and model backend can affect harness reliability.

### 6. 论文的主要结论与发现

- **主基线结果**：
  - AHE 是唯一在验证、ID 和 OOD 上均获得提升的方法（UVG +17.1, IDG +9.1, OODG +6.3）。
  - ACE 获得中等 ID 和 OOD 增益（IDG +3.6, OODG +2.5），验证增益较小（+2.9）。
  - TF-GRPO 验证增益大（+17.1），ID 增益小（+3.6），OOD 为负（-2.5），且 rollout 成本最高。
- **训练回放与遗忘**：
  - AHE 最终回放成功率为 43/80，比初始的 34/80 有净增益；但过程非单调，epoch 4 后出现灾难性的执行路径崩溃（66/80 错误），最终快照恢复。
  - 说明回放诊断揭示了有用的中间快照可能崩溃，最终改进来自修复了运行时消息合约问题。
- **批次大小效应**：
  - 非单调：批次 20 最佳（+17.1 验证，+9.1 ID）；批次 10 和 80 均退化（-14.3、-17.1）。
  - 原因：AHE 更新有固定 token 预算，小批次证据不足且更新频繁，易积累配置风险；大批次证据过载，难以深入分析长尾故障。
- **源多样性**：
  - 混合源（Terminal-Bench + HLE）最终快照保留正向增益；HLE-only 训练中间快照有用但最终崩溃（所有指标为 0）。
  - 单一源导致框架更新集中在验证子系统，最终因消息合约失败而整体瘫痪。
- **跨模型迁移**：
  - 同后端 ID 增益为正（+3.6 到 +9.1），但跨后端不稳定且不对称。
  - 不同训练后端的失败模式不同，导致 AHE 编辑了不同子系统，迁移性依赖于故障面匹配。
  - ID 增益不保证 OOD 增益，甚至负值。
- **整体结论**：自我进化不应简化为单一最终分数；多视图评估（验证、ID、OOD、重放、成本）提供了必要的过程信号，揭示了不同方法的更新机制、泛化能力和稳定性差异。

### 7. 优点

- **系统化的评估框架**：SEAGym 将静态基准转化为动态自我进化环境，通过可配置调度支持多种进化模式，并定义明确的评价视图（更新验证、域内/域外迁移、重放），弥补了现有评估（孤立任务评分或单一序列曲线）的不足。
- **分离任务执行与进化更新**：通过 rollout/update 接口，不同自我进化方法可以保留原生更新规则，同时共享评估协议，提高了可扩展性和公平性。
- **详尽的过程诊断**：保存快照、轨迹引用、更新摘要和合约差异等工件，支持离线分析，从而揭示更新为何提高或降低性能（如运行时合约破坏导致的错误）。
- **丰富的消融实验**：系统调查了批次大小、源多样性、跨模型迁移等维度，证明这些因素显著影响框架可靠性，对实践者具有指导意义。
- **结果互补性**：三种基线方法（AHE、ACE、TF-GRPO）代表了不同的更新范围（框架级、技能级、策略经验级），实验清晰展示了它们的不同行为模式，验证了多视图评估的必要性。

### 8. 不足与局限

- **覆盖范围有限**：仅使用 Terminal-Bench 2.0 和 HLE 两个基准，各覆盖一种主要场景（执行密集型 vs. 推理密集型）。缺少 web/桌面交互、长时软件工程、数据科学工作流、多智能体协作、持续在线任务流等典型智能体应用。
- **基线方法单一**：只对比了三种基于框架/提示更新的方法，没有包括模型权重更新（如在线 RL 微调）、混合系统或其他演化机制。因此结论的泛化性有限。
- **缺乏统计显著性**：大部分实验仅运行一次，没有报告多次运行的平均值和方差。虽然论文列出了一些运行中的故障和恢复，但未系统评估随机性影响。
- **成本评估不完整**：虽然统计了 rollout 和 update token 数以及任务执行时间，但未报告端到端 wall-clock 时间、硬件配置、排队延迟等，难以进行公平的成本比较。
- **OOD 和跨模型分析仍有限**：OOD 只有一种目标域（HLE CS/AI 和工程），跨模型只用了三种较新模型（DeepSeek-V4-Flash、GLM-5.1、GPT-5.4）。更多领域和模型可能揭示不同规律。
- **协议限制**：当前实例化使用固定的 epoch/batch 协议，但调度参数本身可能影响评价结果（如单任务适应 vs. 在线学习 vs. epoch 训练），论文未系统探索不同调度的交互效应。
- **可重复性**：论文提供了详细设置和记录，但未公开代码和任务索引（至少承诺在后续发布），依赖 Harbor 和专有模型，完全复现可能困难。
